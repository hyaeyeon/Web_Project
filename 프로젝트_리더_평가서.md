## 1. 프롬프트 작성 능력 (7.5/10)

### ✅ 강점

#### 1) 역할 부여의 명확성
```
"너는 나의 프로젝트 팀원이 된거야. 같이 아이디어 내주고 코드 작성하는 거 도와줘."
```
- AI에게 명확한 역할을 부여하여 일관된 맥락 유지
- "팀원", "선배" 같은 관계 설정으로 협업적 대화 유도
- **효과**: AI가 단순 답변이 아닌 프로젝트 관점의 조언 제공

#### 2) 제약 조건의 적시 전달
```
"나는 지금 그냥 웹페이지 구현 UI 구현만 하는거야. 백엔드와 연관된 기능은 개발하면 안돼."
```
- 프로젝트의 핵심 제약을 명확히 전달
- 불필요한 제안 사전 차단
- **효과**: AI 응답의 실용성 향상

#### 3) 피드백의 즉시성
```
"화면 잘 떠"
"테스트 저장되고 대시보드에 뜸"
"이제 보여"
```
- 짧고 명확한 상태 피드백
- 빠른 이터레이션 사이클 형성

### ⚠️ 개선 필요 영역

#### 1) 초기 요구사항의 불명확성
- 처음에 "어떤 기능이 있으면 좋을지 말해줘"라고 요청했으나, 구체적인 목표/범위 미설정
- AI가 광범위한 제안을 하게 되어 선택 과정에서 시간 소요
- **개선안**: "12문항 테스트 + 결과 저장 + 대시보드 표시"처럼 초기에 구체적 범위 설정

#### 2) 문제 상황 설명의 모호함
```
"지금 너무 큰 문제가 생겼어. 성향 테스트 결과가 홈 화면에 반영이 안되고, 홈 화면 UI도 조금 바꼈어."
```
- "큰 문제", "조금 바뀜" 같은 주관적 표현
- 구체적인 증상(에러 메시지, 스크린샷 등) 미제공
- **개선안**: "kpiMajor에 '-'만 표시됨, 콘솔에 에러 없음"처럼 구체적 증상 설명

#### 3) 코드 변경 후 확인 부족
- 여러 차례 "코드 수정했어"라고 했으나 실제로 잘못 적용된 경우 발생
- 예: 백틱(`) 대신 따옴표 사용, HTML 구조 잘못 중첩
- **개선안**: 변경 후 diff 확인 또는 주요 부분 재확인 요청

## 2. 프로젝트 관리 능력 (7.0/10)

### ✅ 강점

#### 1) 단계적 진행 의식
```
"지금 Compare화면에 수정해야 할 기능이 있는데 일단 전체 구성을 다 기본적으로 틀 만들고 다시 돌아올까 아니면 지금 바로 수정하는 게 좋을까?"
```
- MVP 우선 완성 후 개선이라는 전략적 사고
- 부분 최적화의 함정 인식

#### 2) 우선순위 판단
- 핵심 기능(테스트 → 대시보드 → 비교 → 로드맵 → FAQ) 순서 결정
- 시각적 요소보다 기능 완성도 우선

### ⚠️ 개선 필요 영역

#### 1) 작업 범위의 잦은 변경
| 시점 | 요청 내용 |
|------|----------|
| 초기 | "기능은 모두 있으면 좋겠어" |
| 중간 | "Filter는 없어도 될 것 같아" |
| 후반 | "로그인/회원가입도 추가하고 싶어" |

- 스코프 크리프(Scope Creep) 경향
- **영향**: 일부 기능이 불완전한 상태로 다음 기능으로 이동
- **개선안**: 초기에 "Must Have / Nice to Have" 구분

#### 2) 테스트 없는 진행
- 각 기능 완성 후 체계적인 테스트 없이 다음 단계로 이동
- 나중에 "결과가 반영이 안 돼" 같은 문제 발생
- **개선안**: 기능별 체크리스트 작성 후 확인

## 3. 의사소통 명확성 (8.0/10)

### ✅ 강점

#### 1) 솔직한 상태 공유
```
"하나도 안바꼈어 그대로야"
"이건 이렇게 두기로 할게. 다음으로 넘어가자"
```
- 문제 발생 시 숨기지 않고 즉시 공유
- 의사결정 후 명확한 방향 전달

#### 2) 오해 발생 시 즉시 정정
```
"너가 좀 오해한 부분이 있는 것 같아. 기존처럼 왼쪽에 네비게이션 바랑 왼쪽 상단의 페이지 로고 등 전체적인 틀 UI는 유지해야 해."
```
- AI의 오해를 발견하고 명확히 정정
- 원하는 결과물에 대한 구체적 설명 추가

### ⚠️ 개선 필요 영역

#### 1) 용어의 비일관성
- "네모 박스", "서브 스킬", "그래프" 등 비표준 용어 혼용
- 같은 요소를 다른 이름으로 지칭하여 혼란 가능성
- **개선안**: 프로젝트 초기에 용어 정의 (예: "Sub Skill 차트 = 막대 그래프")

#### 2) 스크린샷/코드 공유 타이밍
- 문제 상황에서 스크린샷 공유가 늦은 경우 있음
- AI가 여러 가능성을 추측해야 하는 상황 발생
- **개선안**: 문제 발생 시 즉시 시각 자료 + 코드 함께 공유

## 4. 문제 해결 접근법 (7.5/10)

### ✅ 강점

#### 1) 원인 추적 시도
```
"내가 보기에는 index.html 파일에 <div class="chart" id="miniChart"></div> 이게 없는 것 같거든? 확인해줘봐."
```
- 단순히 "안 돼"가 아닌 자체적인 원인 추측 시도
- HTML 구조 문제를 스스로 의심 → 실제로 정확했음

#### 2) 점진적 확인
- "콘솔 에러 있어?" → "styles.css 보여줘" → "HTML 구조 확인해보자"
- 단계적 디버깅 프로세스 학습

### ⚠️ 개선 필요 영역

#### 1) 같은 실수 반복
| 문제 | 발생 횟수 |
|------|----------|
| HTML 태그 미닫힘/중첩 오류 | 3회 이상 |
| 템플릿 리터럴 문법 오류 | 2회 |
| CSS 선택자 미적용 | 2회 |

- **개선안**: 자주 발생하는 실수 체크리스트 작성

#### 2) 디버깅 도구 활용 부족
- 브라우저 개발자 도구(Console, Elements, Application) 적극 활용 미흡
- AI에게 의존하여 원인 파악
- **개선안**: F12 → Console/Elements 먼저 확인하는 습관

## 5. 학습 태도 (8.5/10)

### ✅ 강점

#### 1) 원리 이해 추구
```
"근데 sub skill evaluation 이 나타내는 수치가 뭘 뜻하는거야?"
"min-height 값이 어디있어? 어디서 수정해?"
```
- 단순 복붙이 아닌 의미 파악 시도
- 코드의 역할과 위치에 대한 질문

#### 2) 피드백 수용
- AI의 개선 제안을 대부분 수용
- "그건 이렇게 두기로 할게"처럼 합리적 거절도 함

#### 3) 자기 인식
```
"잠시만 천천히 해볼게"
```
- 자신의 속도와 이해 수준 인식
- 무리하게 진행하지 않고 확인 후 진행

### ⚠️ 개선 필요 영역

#### 1) 문서화 부재
- 학습한 내용(CSS 속성, JS 패턴 등)을 별도로 정리하지 않음
- **영향**: 같은 개념을 여러 번 질문할 가능성
- **개선안**: 핵심 개념/문법 간단 메모

## 7. 성장을 위한 실행 제안

### 단기 (즉시 적용 가능)
1. **변경 후 체크리스트**
   - [ ] HTML 태그 열림/닫힘 확인
   - [ ] 백틱(`) vs 따옴표(') 확인
   - [ ] 브라우저 새로고침 (Ctrl+Shift+R)
   - [ ] 콘솔 에러 확인

2. **문제 보고 템플릿**
   ```
   [증상] 구체적으로 무엇이 안 되는지
   [기대] 원래 어떻게 되어야 하는지
   [시도] 이미 해본 것
   [코드/스크린샷] 관련 부분
   ```

### 중기 (프로젝트 단위)
1. 초기에 기능 범위 문서화 (Must/Nice to Have)
2. 기능별 완료 기준 정의
3. 학습 내용 간단 메모 습관

### 장기 (개발자 성장)
1. 브라우저 개발자 도구 숙달
2. Git 버전 관리로 변경점 추적
3. 단위 테스트 개념 학습
