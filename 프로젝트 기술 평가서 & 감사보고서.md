1.3 코드 품질 상세 분석
✅ 우수 사례 (Best Practices)
CSS 변수 활용 (styles.css): :root에 색상, 그림자, 라운드 값을 정의하여 디자인 시스템을 구축했습니다. 유지보수성과 테마 확장성이 뛰어납니다.
DOM 조작 최적화: document.createDocumentFragment 등을 사용하지는 않았으나, test.js의 renderQuestions 함수처럼 데이터를 기반으로 HTML을 동적 생성하는 로직이 깔끔하게 구현되었습니다.
방어적 코딩 (app.js): renderDashboard 함수 내에서 result가 null일 경우를 처리하는 분기문(if(!result) ... return;)이 잘 작성되어 있어 런타임 에러를 방지하고 있습니다.
⚠️ 개선 필요 사항 (Improvements Needed)
코드 중복 (DRY 위반): 모든 HTML 파일에 <header>와 <aside> 코드가 반복됩니다. 현재는 정적 파일 구조라 어쩔 수 없으나, 추후 Web Components나 간단한 include 함수 도입이 필요합니다.
전역 네임스페이스 오염: STORAGE_KEY 등의 변수가 전역 스코프에 선언되어 있습니다. ES Modules(import/export)를 도입하거나 IIFE(즉시 실행 함수) 패턴으로 캡슐화가 필요합니다.


1.4 UI/UX 평가

2. 코드 감사 보고서 (Code Audit Report)
감사 유형: 보안 및 안정성 감사
감사 대상: 소스코드 전체

2.1 보안 취약점 진단 (Security Audit)

🔴 1. 평문 비밀번호 저장 (Critical)
위치: signup.js (Line 35), login.js (Line 11)
내용: 사용자 비밀번호가 암호화(Hashing)되지 않은 상태로 localStorage에 그대로 저장됩니다.
위험: XSS 공격 등으로 로컬 스토리지 탈취 시 사용자의 비밀번호가 그대로 노출됩니다.
권고: 실제 서비스라면 백엔드 서버에서 bcrypt 등을 사용해야 하며, 프론트엔드 데모라 하더라도 최소한의 해싱(SHA-256 등) 시늉이라도 하거나, "교육용 데모이므로 실제 비밀번호를 사용하지 마세요"라는 경고 문구가 필수적입니다.

🟠 2. 클라이언트 사이드 인증 우회 가능 (High)
위치: login.js
내용: 로그인 체크 로직이 전적으로 클라이언트(localStorage)에 의존합니다.
위험: 사용자가 브라우저 개발자 도구(Application 탭)에서 major_helper_login 키에 임의의 JSON 값을 넣으면 로그인된 상태로 조작 가능합니다.
권고: 중요 데이터가 없는 정보성 사이트라 당장의 피해는 적지만, 인증 구조로서는 부적합함을 인지해야 합니다.

🟡 3. DOM XSS 잠재 위험 (Medium)
위치: faq.js (Line 40), test.js (Line 39)
내용: innerHTML을 사용하여 동적 콘텐츠를 삽입하고 있습니다.
위험: 현재는 내부 데이터(FAQS, QUESTIONS)만 렌더링하므로 안전하지만, 추후 사용자 입력값(예: 검색어, 닉네임 등)을 화면에 뿌려줄 때 이 방식을 유지하면 스크립트 주입 공격에 취약해집니다.
권고: textContent를 사용하거나, DOMPurify 같은 라이브러리 사용을 권장합니다.


2.2 데이터 무결성 및 안정성 (Stability Audit)

🔵 1. 로컬 스토리지 용량 및 데이터 유실
내용: localStorage는 브라우저 캐시 삭제 시 함께 지워집니다. 사용자가 열심히 테스트한 결과나 회원 정보가 영구적이지 않습니다.
권고: 사용자에게 "브라우저 데이터를 삭제하면 기록이 사라집니다"라는 안내가 필요합니다.

🔵 2. 예외 처리 부족
위치: signup.js
내용: JSON.parse 실패 시에 대한 처리가 app.js에는 있으나, 일부 로직에서는 부족할 수 있습니다. 데이터가 오염되었을 때 앱이 멈추지 않도록 초기화 로직(localStorage.clear())을 제안하는 UI가 필요합니다.



2.3 종합 제언 (Recommendations)
보안 강화 (최우선): signup.js와 login.js의 로직을 수정하여 비밀번호를 평문으로 저장하지 않도록 수정하십시오. (교육용이라면 주석으로 위험성 명시 필수)
컴포넌트화: 반복되는 Header/Sidebar HTML 코드를 JavaScript 함수 하나로 렌더링하도록 리팩토링하면 코드 양을 30% 이상 줄일 수 있습니다.
상수 관리: STORAGE_KEY ("major_helper_result_v1")와 같은 상수는 별도의 config.js 파일로 분리하여 관리하는 것이 좋습니다.
